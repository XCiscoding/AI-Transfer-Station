# 项目规划总结

## 1. 项目概述

### 1.1 项目目标

**主要目标：**
构建一个开源、企业级、一站式大模型APIKey管理与调度中枢平台，实现多厂商大模型APIKey的统一纳管、统一出口、统一鉴权、统一监控和统一计费。

**次要目标：**
- 通过虚拟Key隔离真实密钥，降低泄露风险
- 提供智能调度能力，提升可用性、降低成本、保障稳定性
- 支持私有化部署，满足企业合规要求
- 提供完整的监控、告警和审计能力

**成功标准：**
- 支持主流大模型厂商接入（OpenAI、通义、文心、豆包、Claude、Gemini、DeepSeek等）
- 统一接口兼容OpenAI格式，实现零代码迁移
- 平台可用性 ≥ 99.9%
- 单机QPS ≥ 500，接口平均响应 < 200ms
- Docker一键部署成功

### 1.2 目标用户

**主要用户：**
- **超级管理员**：平台运维、权限配置、系统设置，需要全功能管控、安全合规、高可用
- **企业管理员**：团队/项目管理、额度分配、报表查看，需要成本可控、权限隔离、易用

**次要用户：**
- **普通用户/开发者**：调用模型、查看用量，需要稳定调用、简单接入、文档清晰
- **运维角色**：部署维护、监控告警，需要标准化部署、可观测性

**用户场景：**
1. 企业多团队共用AI资源，防止超支、滥用
2. 多模型统一接入，业务无需适配不同厂商接口
3. 密钥集中管理，避免代码硬编码、Git泄露
4. 高可用调度：自动切可用渠道，保障业务不间断
5. 私有化部署：数据不出内网，满足合规要求

## 5. 实现逻辑

### 5.1 数据流

```
用户请求 → Nginx → Spring Boot网关 → 鉴权校验 → 额度校验 → 调度引擎 → 渠道选择 → 厂商API
                ↓
            响应 ← 结果处理 ← 日志记录 ← 额度扣减 ← 调用结果
```

### 5.2 关键算法

**算法1：智能调度算法**
- 目的：在多渠道、多模型条件下选择最优调用路径
- 策略：基于权重、延迟、成本、可用性的综合评分
- 实现：加权轮询 + 健康检查 + 动态权重调整

**算法2：额度实时扣减算法**
- 目的：确保额度扣减的准确性和并发安全
- 策略：Redis预扣 + MySQL持久化 + 异步对账
- 实现：乐观锁 + 分布式锁 + 事务补偿

**算法3：Auto模式智能选择算法**
- 目的：根据角色自动选择最合适的大模型
- 策略：多维度评分 + 策略模式 + 动态权重调整
- 实现：
  1. **角色匹配度评分：** 根据角色-模型映射关系计算匹配度（0-100分）
  2. **性能评分：** 根据模型响应时间、成功率计算性能分（0-100分）
  3. **成本评分：** 根据模型单价计算成本分（0-100分，成本越低分越高）
  4. **可用性评分：** 根据模型当前状态、健康度计算可用性分（0-100分）
  5. **综合评分：** 根据选择策略加权计算总分
     - PERFORMANCE_FIRST: 性能分权重0.5，其他各0.167
     - COST_FIRST: 成本分权重0.5，其他各0.167
     - AVAILABILITY_FIRST: 可用性分权重0.5，其他各0.167
     - BALANCED: 各维度权重0.25
  6. **选择决策：** 选择综合评分最高的模型，如不可用则选择次优模型

### 5.3 安全考虑

**认证：**
- JWT Token认证
- 支持LDAP/SSO集成（P1）
- 登录日志和异常检测

**授权：**
- RBAC权限模型
- 数据范围权限控制
- 最小权限原则

**数据保护：**
- 真实Key AES-256加密存储
- 传输全程HTTPS
- 敏感信息脱敏
- SQL注入和XSS防护

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐
│ 开发者请求  │ →  │ 网关接收请求 │ →  │ Key鉴权校验 │ →  │ 额度/限流校验│
│ (虚拟Key)   │    │              │    │             │    │              │
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘
                                                                ↓
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐
│ 返回结果    │ ←  │ 日志记录     │ ←  │ 额度扣减    │ ←  │ 调度最优渠道 │
│             │    │              │    │             │    │              │
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘
                                                                ↓
                                                         ┌──────────────┐
                                                         │ 转发到厂商API│
                                                         │              │
                                                         └──────────────┘
```

**异常处理：**
- Key无效 → 返回401错误
- 额度不足 → 返回403错误
- 限流触发 → 返回429错误
- 渠道不可用 → 自动重试/切渠道 → 返回503错误

#### 流程2：渠道故障转移流程

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐
│ 渠道请求失败│ →  │ 触发重试机制 │ →  │ 仍失败      │ →  │ 标记渠道不健康│
│             │    │ (可配置次数) │    │             │    │              │
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘
                                                                ↓
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐
│ 恢复上线    │ ←  │ 健康检查恢复 │ ←  │ 自动禁用    │ ←  │ 切换备用渠道 │
│             │    │              │    │ 并告警      │    │              │
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘
```

```
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘
                                                                ↓
                                                         ┌──────────────┐
```

## 13. 开发规范

### 13.1 代码规范

**后端（Java）：**
- 遵循阿里巴巴Java开发手册
- 使用Lombok简化代码
- 统一异常处理
- 统一返回结果格式
- 方法注释覆盖率100%

**前端（JavaScript/Vue）：**
- 遵循Vue官方风格指南
- 使用ESLint + Prettier格式化
- 组件命名采用PascalCase
- Props定义使用对象形式
- 方法注释覆盖率80%+

### 13.2 数据库规范

**技术要点：**
- 使用MySQL 8.0+特性
- 表名使用下划线命名法
- 必须包含created_at、updated_at字段
- 敏感字段（如真实Key）使用TEXT类型存储加密数据

**核心表设计：**
1. **用户与权限表：** users, roles, permissions, user_roles, role_permissions
2. **渠道与模型表：** channels, models, channel_models
3. **APIKey表：** real_keys, virtual_keys
4. **额度与计费表：** quotas, billing_rules, quota_transactions
5. **日志表：** call_logs, login_logs
6. **团队与项目表：** teams, projects, team_members
7. **Auto模式表：** role_model_mappings, model_selection_logs

**Auto模式相关表设计：**

**role_model_mappings（角色模型映射表）：**
- id: 主键
- role_id: 角色ID（关联roles表）
- role_name: 角色名称（冗余字段，便于查询）
- recommended_models: 推荐模型列表（JSON格式，包含模型ID、优先级、权重）
- selection_strategy: 选择策略（PERFORMANCE_FIRST/COST_FIRST/AVAILABILITY_FIRST/BALANCED）
- fallback_models: 降级模型列表（JSON格式）
- is_enabled: 是否启用
- priority: 优先级
- created_at: 创建时间
- updated_at: 更新时间

**model_selection_logs（模型选择日志表）：**
- id: 主键
- call_id: 调用ID（关联call_logs表）
- user_id: 用户ID
- role_id: 角色ID
- role_name: 角色名称
- requested_model: 请求的模型（可能为auto）
- selected_model: 选中的模型
- selection_reason: 选择理由（JSON格式，包含评分、决策过程）
- selection_strategy: 使用的选择策略
- candidate_models: 候选模型列表（JSON格式）
- is_fallback: 是否使用降级模型
- response_time: 选择耗时（毫秒）
- created_at: 创建时间

### 13.3 接口规范

- RESTful API设计
- 统一返回格式：`{code, message, data}`
- 统一错误码定义
- 接口版本管理：`/api/v1/`
- 接口文档使用Swagger

**Auto模式相关API接口：**

**1. 角色模型映射管理接口：**
- `GET /api/v1/auto/mappings` - 查询映射规则列表
- `POST /api/v1/auto/mappings` - 创建映射规则
- `PUT /api/v1/auto/mappings/{id}` - 更新映射规则
- `DELETE /api/v1/auto/mappings/{id}` - 删除映射规则
- `PUT /api/v1/auto/mappings/{id}/priority` - 调整映射规则优先级
- `PUT /api/v1/auto/mappings/{id}/strategy` - 配置选择策略

**2. 模型选择接口：**
- `POST /api/v1/auto/select` - 智能选择模型（内部接口）
- `GET /api/v1/auto/select/logs` - 查询选择日志
- `GET /api/v1/auto/select/statistics` - 查询选择统计
- `GET /api/v1/auto/select/accuracy` - 查询选择准确性分析

**3. Auto模式调用接口：**
- `POST /v1/chat/completions` - Auto模式聊天接口（model参数传"auto"）
- `POST /v1/embeddings` - Auto模式嵌入接口（model参数传"auto"）

**接口请求示例：**
```json
{
  "model": "auto",
  "messages": [
    {"role": "user", "content": "请帮我写一个Java方法"}
  ]
}
```
