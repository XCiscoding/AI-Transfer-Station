# AI IDE自动化工作流规范

## 概述

本文档定义AI IDE执行项目开发任务的自动化工作流规范。它提供了任务执行、测试和上下文管理的标准化流程。

## 工作流定义

### 核心工作流

```
┌─────────────────────────────────────────────────────────────────┐
│                    AI IDE自动化工作流                            │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
                    ┌───────────────────────┐
                    │   1. 读取tasks.md     │
                    │   获取下一个任务      │
                    └───────────────────────┘
                                │

                    │   2. 执行任务         │
                    │   实现代码            │
                    └───────────────────────┘
```

## 详细工作流步骤

### 步骤1：任务选择

**动作：** 读取tasks.md并选择下一个可用任务

**规则：**
1. 按优先级顺序选择任务：高 → 中 → 低
2. 遵守任务依赖关系（先完成前置任务）
3. 只选择状态为"未开始"的任务
4. 在progress.md中更新任务状态为"进行中"

**输出：**
- 任务ID
- 任务描述
- 验收标准
- 技术要点

### 步骤2：任务执行

**动作：** 根据规范实现任务

**流程：**
1. 分析任务需求
2. 设计实现方案
3. 编写/修改代码
4. 遵循项目编码规范
5. 添加必要的文档

**指南：**
- 一次专注一个任务
- 遵循提供的技术要点
- 确保代码质量
- 保持与现有代码库的一致性

### 步骤3：测试

**动作：** 验证实现是否满足验收标准

**测试类型：**
1. **单元测试：** 测试单个组件
2. **集成测试：** 测试组件交互
3. **功能测试：** 测试功能特性
4. **手动测试：** 验证面向用户的行为

**测试执行：**
```bash
# 运行单元测试
mvn test

# 运行集成测试
mvn verify

# 运行所有测试
mvn clean install
```

### 步骤4A：测试通过

**动作：** 标记任务为已完成并更新进度

**步骤：**
1. 更新progress.md：
   - 将状态改为 🟢（已完成）
   - 设置进度为100%
   - 设置结束日期
   - 记录实际工时
2. 更新tasks.md：
   - 将状态改为"已完成"
3. 记录重要备注

### 步骤4B：测试失败

**动作：** 分析问题并调试

**调试流程：**
1. **识别问题：**
   - 查看错误消息
   - 检查测试输出
   - 分析日志

2. **根因分析：**
   - 追踪错误来源
   - 识别出了什么问题
   - 记录发现

3. **修复实施：**
   - 应用必要的修复
   - 重新运行测试
   - 验证修复有效

4. **在promote.md中记录：**
   - 创建Bug修复记录
   - 记录根因
   - 记录解决方案

5. **重新测试：**
   - 再次运行测试
   - 如果通过 → 转到步骤4A
   - 如果失败 → 重复调试流程

### 步骤5：清除上下文

**动作：** 为下一个任务重置AI IDE上下文

**为什么清除上下文：**
- 防止上下文污染
- 保持对新任务的专注
- 避免携带无关信息
- 确保每个任务的清洁状态

**清除什么：**
- 前一个任务的详情
- 临时变量
- 调试信息
- 中间状态

**保留什么：**
- 项目结构知识
- 编码规范
- 配置设置
- 必要的依赖

### 步骤6：获取下一个任务

**动作：** 返回步骤1并选择下一个任务

**流程：**
1. 读取tasks.md
2. 找到下一个可用任务
3. 验证依赖已满足
4. 开始新的任务循环

## 状态管理

### 任务状态

| 状态 | 符号 | 描述 |
|------|------|------|
| 未开始 | ⚪ | 任务尚未开始 |
| 进行中 | 🔵 | 任务正在进行 |
| 已完成 | 🟢 | 任务已完成并测试 |
| 阻塞 | 🔴 | 任务无法继续 |
| 暂停 | 🟡 | 任务暂时暂停 |

### 状态转换

```
⚪ 未开始
    ↓ (开始任务)
🔵 进行中
    ↓ (完成)          ↓ (阻塞)
🟢 已完成          🔴 阻塞
                              ↓ (解除阻塞)
                          🔵 进行中
```

## 错误处理

### 常见错误场景

#### 1. 依赖未满足
**场景：** 任务需要另一个未完成的任务

**解决方案：**
- 检查任务依赖
- 先完成前置任务
- 依赖解决后重试

#### 2. 测试环境问题
**场景：** 测试因环境问题失败

**解决方案：**
- 检查环境配置
- 验证依赖已安装
- 检查数据库连接
- 查看环境变量

#### 3. 代码冲突
**场景：** 变更与现有代码冲突

**解决方案：**
- 查看现有代码结构
- 小心解决冲突
- 确保向后兼容
- 更新受影响的组件

#### 4. 信息不足
**场景：** 任务描述不清楚或不完整

**解决方案：**
- 查看summary.md获取上下文
- 检查相关任务
- 记录假设
- 用最佳判断继续

## 进度跟踪

### 自动更新

AI IDE应自动更新：

1. **progress.md：**
   - 任务状态变更
   - 进度百分比
   - 时间跟踪
   - 备注和评论

2. **tasks.md：**
   - 任务状态更新
   - 完成标记

3. **promote.md：**
   - Bug修复记录
   - 优化记录
   - 改进备注

### 手动检查点

在每个任务完成时，验证：

- [ ] 代码已实现
- [ ] 测试通过
- [ ] 文档已更新
- [ ] 进度已记录
- [ ] 上下文已清除

## 最佳实践

### AI IDE

1. **一次一个任务：**
   - 永远不要同时处理多个任务
   - 在开始下一个之前完成当前任务

2. **充分测试：**
   - 运行所有相关测试
   - 验证验收标准
   - 检查边界情况

3. **记录一切：**
   - 记录所有变更
   - 记录遇到的任何问题
   - 更新相关文档

4. **正确清除上下文：**
   - 每个任务后重置
   - 保留必要知识
   - 为新任务全新开始

5. **遵循优先级顺序：**
   - 高优先级优先
   - 遵守依赖关系
   - 保持逻辑流程

### 开发者

1. **审查生成的计划：**
   - 验证任务拆解
   - 检查验收标准
   - 验证估算

2. **监控进度：**
   - 定期检查progress.md
   - 查看已完成任务
   - 尽早识别阻塞

3. **提供反馈：**
   - 如需更新任务描述
   - 添加缺失的验收标准
   - 记录额外需求

## 工作流配置

### 可自定义参数

| 参数 | 默认值 | 描述 |
|------|--------|------|
| max_retries | 3 | 最大调试重试次数 |
| test_timeout | 300 | 测试执行超时（秒） |
| context_clear | true | 任务后自动清除上下文 |
| auto_progress | true | 自动更新progress.md |

### 配置示例

```json
{
  "workflow": {
    "max_retries": 3,
    "test_timeout": 300,
    "context_clear": true,
    "auto_progress": true,
    "priority_order": ["高", "中", "低"]
  }
}
```

# MVP阶段专项工作流规范（追加）

## 概述

本文档为MVP阶段（13个核心任务）提供专项工作流指导，基于原通用工作流规范，针对MVP范围进行细化和补充。

**适用范围：**
- M001 系统基础模块（登录认证）
- M002 渠道管理
- M003 APIKey管理

**总任务数：** 13个
**预估工时：** 41小时
**目标周期：** 约9个工作日

## MVP阶段特殊规则

### 1. 任务执行顺序约束

**强制依赖链（必须遵守）：**

```
TASK-M001-002 (后端搭建)
    ↓
TASK-M001-004 (实体) ← TASK-M001-003 (前端搭建) [可并行]
    ↓
TASK-M001-005 (JWT认证)
    ↓
TASK-M001-006 (登录接口)
    ↓ 并行
TASK-M001-007 (登录页面) + TASK-M002-001 (渠道实体)
    ↓
TASK-M002-002 (渠道接口)
    ↓
TASK-M002-003 (渠道页面)
    ↓
TASK-M003-001 (真实Key实体) + TASK-M003-003 (虚拟Key实体) [可并行]
    ↓
TASK-M003-002 (真实Key接口)
    ↓
TASK-M003-004 (虚拟Key接口)
    ↓
TASK-M003-005 (APIKey管理页面)
```

### 2. MVP阶段质量门禁

#### 后端任务完成后必须验证：

- [ ] **编译通过**: `mvn clean compile` 无错误
- [ ] **单元测试**: 核心逻辑覆盖率 ≥ 60%
- [ ] **接口文档**: Swagger注解完整，可访问 `/swagger-ui.html`

- [ ] **构建成功**: `npm run build` 无错误

### 4. MVP阶段代码审查要点

#### 后端代码审查清单：

**安全相关：**
- [ ] 密码字段使用 `@JsonIgnore` 或不序列化
- [ ] SQL注入防护（使用JPA参数化查询）
- [ ] XSS防护（输出转义，Spring Boot默认支持）
- [ ] AES密钥不从硬编码读取（使用配置文件或环境变量）
- [ ] JWT Secret足够长（≥256 bits）

**性能相关：**
- [ ] 数据库查询使用分页（避免全表扫描）
- [ ] N+1查询问题（使用JOIN FETCH或DTO投影）
- [ ] 大文本字段延迟加载（`@ManyToOne(fetch = LAZY)`）
- [ ] 合理使用缓存（Redis）

**代码质量：**
- [ ] 遵循RESTful API设计规范
- [ ] 统一异常处理（GlobalExceptionHandler）
- [ ] 日志级别合理（debug/info/warn/error）
- [ ] 方法注释清晰（Javadoc）

#### 前端代码审查清单：

**用户体验：**
- [ ] 加载状态显示（v-loading指令）
- [ ] 操作确认（删除等危险操作）
- [ ] 表单反馈及时（实时校验）
- [ ] 错误信息友好（ElMessage）

**代码规范：**
- [ ] 组件命名语义化（PascalCase）
- [ ] Props定义明确（对象形式）
- [ ] 事件命名统一（handleXxx）
- [ ] CSS作用域隔离（scoped）

### 5. MVP阶段上下文管理规则

#### 任务切换时的上下文保留清单：

**必须保留（跨任务）：**
- 项目结构和技术栈理解
- 数据库schema.sql表结构
- 统一返回格式Result.java的定义
- Axios封装和拦截器配置
- 路由守卫逻辑
- 用户状态管理store设计

**必须清除（防污染）：**
- 上一个任务的临时变量
- 具体业务实现细节（除非有关联）
- 调试信息和中间状态
- 未解决的Bug记录（应记入promote.md）

#### 上下文传递模板：

```markdown
## 上下文交接备忘

### 已完成
- [任务ID]: [简述完成内容和关键决策]

### 关键文件路径
- [重要文件路径及说明]

### 待注意事项
- [下一任务需要注意的点]

### 已知限制
- [当前实现的已知局限]
```

### 6. MVP阶段失败处理机制

#### 失败分类与应对策略：

| 失败类型 | 判定标准 | 应对策略 |
|----------|----------|----------|
| 编译失败 | mvn/npm build 报错 | 检查依赖、语法、配置，1次修复机会 |
| 测试失败 | 单元测试未通过 | 分析失败原因，定位Bug，2次修复机会 |
| 功能缺陷 | 验收标准未满足 | 返回项目总经理评估是否阻塞 |
| 性能不达标 | 响应时间超标 | 记录问题，MVP阶段可暂缓优化 |
| 安全漏洞 | 敏感数据泄露 | **立即停止**，返回项目总经理 |

#### 熔断规则（MVP阶段强化）：

1. **同一任务连续2次失败**：
   - 停止当前任务
   - 记录详细失败日志到promote.md
   - 返回项目总经理重新评估任务拆分

2. **同一模块连续3个任务受阻**：
   - 进入模块级BLOCKED状态
   - 召集技术评审会议
   - 可能需要调整技术方案或降低验收标准

3. **关键路径阻塞超过4小时**：
   - 触发风险升级流程
   - 考虑并行开发其他非依赖任务
   - 评估是否需要调整MVP范围

### 9. MVP阶段特殊技术约定

#### 接口版本管理：

MVP阶段所有API使用固定前缀：
```
/api/v1/auth/*      # 认证相关
/api/v1/channels/*  # 渠道管理
/api/v1/real-keys/* # 真实Key管理
/api/v1/virtual-keys/* # 虚拟Key管理
```

#### 数据库初始化：

MVP阶段假设schema.sql已预先执行，应用启动时不自动建表：
```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: validate  # 仅验证，不自动建表
```

#### 配置管理：

敏感配置使用环境变量或配置中心（MVP阶段使用application.yml占位符）：
```yaml
# 必须修改的配置项（生产环境）
jwt:
  secret: ${JWT_SECRET:your-dev-secret-key}
aes:
  secret-key: ${AES_SECRET:your-dev-aes-key}
spring:
  datasource:
    password: ${DB_PASSWORD:root}
  data:
    redis:
      password: ${REDIS_PASSWORD:}
```

#### 日志规范：

MVP阶段日志级别建议：
```yaml
logging:
  level:
    com.aikey: debug        # 开发环境调试
    org.springframework.security: info
    org.hibernate.SQL: debug  # 显示SQL（开发环境）
```

### 10. MVP阶段回退预案

#### 场景1: 时间严重超期（超过预计50%）

**应对措施：**
1. 暂停非关键功能（如连通性测试异步化、批量导入）
2. 降低部分验收标准（如去掉Swagger注解完整性要求）
3. 将复杂UI简化为基础表格+表单
4. 记录所有降级决策，后续补齐

#### 场景2: 技术方案不可行

**应对措施：**
1. 立即停止当前任务
2. 回退到上一个稳定版本
3. 重新评估技术选型
4. 寻找替代方案（如简化加密算法、更换UI组件库）

#### 场景3: 核心需求变更

**应对措施：**
1. 冻结当前MVP范围
2. 将新需求纳入下一迭代
3. 评估对已完成任务的影响
4. 必要时重新规划剩余任务
