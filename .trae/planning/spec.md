# AI IDE自动化工作流规范

## 概述

本文档定义AI IDE执行项目开发任务的自动化工作流规范。它提供了任务执行、测试和上下文管理的标准化流程。

## 详细工作流步骤

### 步骤1：任务选择

**动作：** 读取tasks.md并选择下一个可用任务

**规则：**
1. 按优先级顺序选择任务：高 → 中 → 低
2. 遵守任务依赖关系（先完成前置任务）
3. 只选择状态为"未开始"的任务
4. 在progress.md中更新任务状态为"进行中"

**输出：**
- 任务ID
- 任务描述
- 验收标准
- 技术要点

### 步骤2：任务执行

**动作：** 根据规范实现任务

**流程：**
1. 分析任务需求
2. 设计实现方案
3. 编写/修改代码
4. 遵循项目编码规范
5. 添加必要的文档

**指南：**
- 一次专注一个任务
- 遵循提供的技术要点
- 确保代码质量
- 保持与现有代码库的一致性

### 步骤3：测试

**动作：** 验证实现是否满足验收标准

**测试类型：**
1. **单元测试：** 测试单个组件
2. **集成测试：** 测试组件交互
3. **功能测试：** 测试功能特性
4. **手动测试：** 验证面向用户的行为

**测试执行：**
```bash
# 运行单元测试
mvn test

# 运行集成测试
mvn verify

# 运行所有测试
mvn clean install
```

### 步骤4A：测试通过

**动作：** 标记任务为已完成并更新进度

**步骤：**
1. 更新progress.md：
   - 将状态改为 🟢（已完成）
   - 设置进度为100%
   - 设置结束日期
   - 记录实际工时
2. 更新tasks.md：
   - 将状态改为"已完成"
3. 记录重要备注

### 步骤4B：测试失败

**动作：** 分析问题并调试

**调试流程：**
1. **识别问题：**
   - 查看错误消息
   - 检查测试输出
   - 分析日志

2. **根因分析：**
   - 追踪错误来源
   - 识别出了什么问题
   - 记录发现

3. **修复实施：**
   - 应用必要的修复
   - 重新运行测试
   - 验证修复有效

4. **在promote.md中记录：**
   - 创建Bug修复记录
   - 记录根因
   - 记录解决方案

5. **重新测试：**
   - 再次运行测试
   - 如果通过 → 转到步骤4A
   - 如果失败 → 重复调试流程

### 步骤5：清除上下文

**动作：** 为下一个任务重置AI IDE上下文

**为什么清除上下文：**
- 防止上下文污染
- 保持对新任务的专注
- 避免携带无关信息
- 确保每个任务的清洁状态

**清除什么：**
- 前一个任务的详情
- 临时变量
- 调试信息
- 中间状态

**保留什么：**
- 项目结构知识
- 编码规范
- 配置设置
- 必要的依赖

### 步骤6：获取下一个任务

**动作：** 返回步骤1并选择下一个任务

**流程：**
1. 读取tasks.md
2. 找到下一个可用任务
3. 验证依赖已满足
4. 开始新的任务循环

## 状态管理

### 任务状态

| 状态 | 符号 | 描述 |
|------|------|------|
| 未开始 | ⚪ | 任务尚未开始 |
| 进行中 | 🔵 | 任务正在进行 |
| 已完成 | 🟢 | 任务已完成并测试 |
| 阻塞 | 🔴 | 任务无法继续 |
| 暂停 | 🟡 | 任务暂时暂停 |

### 状态转换

```
⚪ 未开始
    ↓ (开始任务)
🔵 进行中
    ↓ (完成)          ↓ (阻塞)
🟢 已完成          🔴 阻塞
                              ↓ (解除阻塞)
                          🔵 进行中
```

## 错误处理

### 常见错误场景

#### 1. 依赖未满足
**场景：** 任务需要另一个未完成的任务

**解决方案：**
- 检查任务依赖
- 先完成前置任务
- 依赖解决后重试

#### 2. 测试环境问题
**场景：** 测试因环境问题失败

**解决方案：**
- 检查环境配置
- 验证依赖已安装
- 检查数据库连接
- 查看环境变量

#### 3. 代码冲突
**场景：** 变更与现有代码冲突

**解决方案：**
- 查看现有代码结构
- 小心解决冲突
- 确保向后兼容
- 更新受影响的组件

#### 4. 信息不足
**场景：** 任务描述不清楚或不完整

**解决方案：**
- 查看summary.md获取上下文
- 检查相关任务
- 记录假设
- 用最佳判断继续

## 进度跟踪

### 自动更新

AI IDE应自动更新：

1. **progress.md：**
   - 任务状态变更
   - 进度百分比
   - 时间跟踪
   - 备注和评论

2. **tasks.md：**
   - 任务状态更新
   - 完成标记

3. **promote.md：**
   - Bug修复记录
   - 优化记录
   - 改进备注

### 手动检查点

在每个任务完成时，验证：

- [ ] 代码已实现
- [ ] 测试通过
- [ ] 文档已更新
- [ ] 进度已记录
- [ ] 上下文已清除

## 最佳实践

### AI IDE

1. **一次一个任务：**
   - 永远不要同时处理多个任务
   - 在开始下一个之前完成当前任务

2. **充分测试：**
   - 运行所有相关测试
   - 验证验收标准
   - 检查边界情况

3. **记录一切：**
   - 记录所有变更
   - 记录遇到的任何问题
   - 更新相关文档

4. **正确清除上下文：**
   - 每个任务后重置
   - 保留必要知识
   - 为新任务全新开始

5. **遵循优先级顺序：**
   - 高优先级优先
   - 遵守依赖关系
   - 保持逻辑流程

### 开发者

1. **审查生成的计划：**
   - 验证任务拆解
   - 检查验收标准
   - 验证估算

2. **监控进度：**
   - 定期检查progress.md
   - 查看已完成任务
   - 尽早识别阻塞

3. **提供反馈：**
   - 如需更新任务描述
   - 添加缺失的验收标准
   - 记录额外需求

## 工作流配置

### 可自定义参数

| 参数 | 默认值 | 描述 |
|------|--------|------|
| max_retries | 3 | 最大调试重试次数 |
| test_timeout | 300 | 测试执行超时（秒） |
| context_clear | true | 任务后自动清除上下文 |
| auto_progress | true | 自动更新progress.md |

### 配置示例

```json
{
  "workflow": {
    "max_retries": 3,
    "test_timeout": 300,
    "context_clear": true,
    "auto_progress": true,
    "priority_order": ["高", "中", "低"]
  }
}
```

## 项目特定工作流

### 后端开发工作流

```
1. 创建实体类（Entity）
   ↓
2. 创建Repository接口
   ↓
3. 创建DTO类
   ↓
4. 创建Service接口和实现
   ↓
5. 创建Controller
   ↓
6. 编写单元测试
   ↓
7. 编写集成测试
   ↓
8. 更新API文档
```

### 前端开发工作流

```
1. 创建页面组件（.vue文件）
   ↓
2. 创建API接口文件
   ↓
3. 实现页面布局
   ↓
4. 实现业务逻辑
   ↓
5. 添加表单验证
   ↓
6. 实现错误处理
   ↓
7. 测试页面功能
   ↓
8. 优化用户体验
```

### 数据库开发工作流

```
1. 设计表结构
   ↓
2. 编写DDL脚本
   ↓
3. 创建索引
   ↓
4. 插入初始数据
   ↓
5. 测试SQL脚本
   ↓
6. 更新文档
```
