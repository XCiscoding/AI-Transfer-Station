## 文档版本

- **创建日期**: 2026-04-02
- **创建者**: 项目总经理
- **生效时间**: 立即生效（从TASK-M003-002开始强制执行）

## 一、背景与问题

### 历史问题（已修复）
在MVP开发的M001和M002模块阶段（共8个已完成任务），仅有2个任务生成了执行日志，日志覆盖率为**25%**。

### 根因分析（5 Whys结论）
1. **流程缺陷**: 调度指令未包含"必须生成执行日志"的强约束
2. **意识偏差**: 将日志视为可选附加品而非核心交付物
3. **机制缺失**: 无日志完整性检查环节（交付验收漏项）
4. **监督不力**: 总经理接收交付时未核查日志存在性

### 影响评估
| 缺失日志的后果 | 严重程度 |
|---------------|---------|
| 问题追溯困难（无证据链） | 🔴 致命 |
| 知识传承断裂（新人无法了解历史决策） | 🔴 致命 |
| 质量改进缺乏数据支撑 | 🟠 严重 |
| 审计合规不满足要求 | 🟠 严重 |
| 项目资产管理混乱 | 🟡 中等 |

## 二、核心原则

### 原则1: 日志即交付物
**开发日志不是可选的附加品，而是每个任务的强制性交付物，与代码具有同等重要性。**

### 原则2: 即时生成
**日志必须在任务完成的同一轮次内即时生成，不得延后补录。**

### 原则3: 完整记录
**日志必须包含完整的执行过程、产出清单、问题单、技术亮点和验收检查。**

### 原则4: 总经理兜底
**项目总经理在接收任何Agent交付时，必须将"执行日志是否存在且完整"作为验收前置条件。**

---

## 三、强制执行规则

### 规则1: 调度指令必含日志要求
每次调度专业Agent时，调度指令**必须**明确包含以下内容：

```
## 交付要求（强制）
1. [代码产出] 完成XX文件的编写/修改
2. [编译验证] 通过mvn compile或IDE编译验证
3. ⭐[执行日志] 必须生成PROGRESS-{TaskID}-execution-log.md到.progress/目录
   - 日志模板见本SOP第四章节
   - 日志必须在任务完成时同步提交
   - 缺失日志视为交付不完整，不予通过验收
```

### 规则2: Agent交付必含日志字段
专业Agent的结构化交付中**必须**包含：

```json
{
  "goal": "本轮目标",
  "execution": "实际执行过程",
  "result": {
    "code_files": ["文件列表"],
    "compile_status": "成功/失败",
    "log_file": ".trae/progress/PROGRESS-{TaskID}-execution-log.md",  // ← 强制字段
    "issues": ["问题单列表"]
  },
  "unfinished": "未完成项",
  "risk": "风险/问题",
  "next_step": "建议下一步",
  "confidence": "high/medium/low"
}
```

**关键字段**: `result.log_file` — 如果此字段为空或null，**交付直接拒绝**。

### 规则3: 总经理验收必查日志
项目总经理接收交付时的**验收检查清单**（新增项用⭐标注）：

| 序号 | 检查项 | 通过标准 | 不通过处理 |
|------|--------|----------|-----------|
| 1 | 代码文件存在性 | Glob可找到所有声明的新建/修改文件 | 要求补充 |
| 2 | 编译验证结果 | BUILD SUCCESS 或 class文件已生成 | 返回Debug |
| 3 | ⭐**日志文件存在性** | `.trae/progress/PROGRESS-{TaskID}-execution-log.md` 存在 | **拒绝交付，要求补录** |
| 4 | ⭐**日志完整性** | 包含第五章节要求的全部9个Section | **拒绝交付，要求补全** |
| 5 | 问题单记录 | 如有问题需在promote/有对应ISSUE文件 | 要求补充 |
| 6 | 状态一致性 | task.md状态与实际一致 | 更新状态 |

**熔断机制**: 
- 连续2次因日志缺失被拒绝 → 调度指令增加加粗警告
- 连续3次 → 升级为BLOCKED，需人工介入审查流程

## 四、日志模板规范

### 文件命名规范
```
PROGRESS-{任务ID}-execution-log.md
示例: PROGRESS-M003-002-execution-log.md
```

### 存储位置
```
项目根目录/.trae/progress/
```

### 强制包含的章节（9个Section）

#### Section 1: 任务信息（必需）
```markdown
## 任务信息
- **任务ID**: TASK-XXXX-NNN
- **任务名称**: [任务标题]
- **所属模块**: [模块名]
- **执行时间**: YYYY-MM-DD
- **责任角色**: [角色名称]
- **调度记录**: #NNN
- **预估工时**: X小时
- **依赖**: [依赖任务] ✅/⏳
```

#### Section 2: 执行过程（必需）
```markdown
## 执行过程

### 阶段N: [阶段名称] ✅/❌
**操作内容**:
1. ✅/❌ [具体操作步骤]
...
```

**至少包含以下阶段**:
- 需求分析与上下文读取
- 代码生成（分文件列出）
- 编译验证（含错误详情如有）

#### Section 3: 交付物清单（必需）
```markdown
## 交付物清单

### 新建文件 (N个)
| 文件 | 行数 | 功能 | 状态 |
|------|------|------|------|

### 修改文件 (N个)
| 文件 | 修改内容 | 原因 | 状态 |
```

#### Section 4: 问题单关联（必需）
```markdown
## 问题单关联
- [ISSUE-XXX](链接): [问题描述] ([High/Medium/Low], 已修复/待修复)
✅ **无问题单**  // 如果没有问题
```

#### Section 5: 技术亮点实现（必需）
```markdown
## 技术亮点实现
1. ✅ [亮点1]
2. ✅ [亮点2]
...
```

#### Section 6: 验收标准检查清单（必需）
```markdown
## 验收标准检查清单
- [x] [标准1]
- [x] [标准2]
...
```

**必须逐项勾选，不允许省略**

#### Section 7: 最终状态（必需）
```markdown
## 最终状态
✅ **TASK-XXXX-NNN 已完成**

**总产出**: 
- 新增N个源文件
- 修改N个已有文件
- 发现并修复N个缺陷
- 项目累计：X个源文件，全部编译通过
```

#### Section 8: 经验总结（推荐但非强制）
```markdown
## 经验总结（正面案例/教训）
1. **[经验标题]**: [详细描述]
...
```

#### Section 9: 日志元信息（必需）
```markdown
---
*本日志由[角色]于YYYY-MM-DD[创建/补录]*
```

**如果标注"补录"，必须说明原流程遗漏原因**

## 五、违规处理流程

### 场景1: Agent交付时缺少日志
```
检测: result.log_file 为空或文件不存在
→ 总经理输出拒绝通知:
  ❌ 交付不完整：缺少执行日志
  → 要求Agent立即补录
  → 记录到promote/LOG-VIOLATION-{序号}.md
  → 不允许进入下一任务
```

### 场景2: 总经理验收时忘记检查日志
```
检测: 后续审计发现日志缺失
→ 总经理自我问责:
  → 补录日志（标注"补录"）
  → 创建LOG-VIOLATION文档
  → 分析遗漏原因
  → 更新本SOP防止重演
```

### 场景3: 连续违反日志规定
```
第1次: 警告 + 要求补录
第2次: 加粗警告 + 调度指令强化
第3次: BLOCKED + 人工审查流程
```

## 六、质量指标

### 目标KPI（从2026-04-02起统计）
| 指标 | 当前值 | 目标值 | 测量方式 |
|------|--------|--------|----------|
| 日志覆盖率 | 100% (7/7) | **保持100%** | 已完成任务数 / 有日志任务数 |
| 日志及时率 | 待统计 | **≥95%** | 即时生成数 / 总日志数（不含历史补录） |
| 日志完整率 | 待统计 | **100%** | 包含9个Section的日志数 / 总日志数 |
| 平均日志行数 | 待统计 | **≥150行** | 所有日志总行数 / 日志数量 |

## 七、工具支持（可选增强）

### 可选: 日志自动检查脚本
```bash
# check-log-completeness.sh (伪代码)
for task in completed_tasks:
  log_file = f".trae/progress/PROGRESS-{task}-execution-log.md"
  if not exists(log_file):
    VIOLATION_COUNT += 1
    print(f"❌ {task}: 缺少日志")
  else:
    sections = count_sections(log_file)
    if sections < 9:
      VIOLATION_COUNT += 1
      print(f"⚠️ {task}: 日志不完整({sections}/9 sections)")
    else:
      print(f"✅ {task}: 日志完整")
```

### 可选: IDE插件提示
在调度指令模板中嵌入提醒：
> ⚠️ **重要提醒**: 任务完成后必须生成执行日志到 .trae/progress/ 目录！详见 `开发日志强制生成机制SOP.md`
