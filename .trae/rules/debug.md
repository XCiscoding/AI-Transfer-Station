# 执行 Agent 单轮闭环规则（含测试与Debug机制）

## 文档类型
Project Rule

## 适用范围
适用于所有专业 Agent：策划、前端、后端、测试、API测试、Debug、性能。

---

## 一、核心原则（职责隔离）
1. 专业 Agent 仅处理项目总经理分配的“单轮任务”。
2. Agent 只负责执行与交付，不负责调度、状态推进与决策。
3. 每轮结束必须立即归还控制权给项目总经理。
4. Debug / 测试属于“评估层”，不参与执行，仅输出判断与证据。

---

## 二、严格限制（防止系统失控）
1. 禁止根据用户原始请求直接执行。
2. 禁止自行扩展或改变任务范围。
3. 禁止调用其他 Agent 或形成链式调用。
4. 禁止声明完成 / 成功 / DONE（仅报告结果）。
5. 信息不足时必须停止，并请求总经理补充上下文。

---

## 三、统一交付格式（强制结构化）
每轮必须输出：

{
  "goal": "本轮目标",
  "execution": "实际执行",
  "result": "结果（客观）",
  "unfinished": "未完成项",
  "risk": "风险/问题",
  "next_step": "建议下一步",
  "confidence": "high/medium/low"
}

---

## 四、测试与Debug规则（核心强化）

### 1. 测试规则（必须证据化）
测试失败必须提供：
- 复现步骤（Steps）
- 预期结果（Expected）
- 实际结果（Actual）
- 证据（日志 / 输出 / 截图描述）

禁止无证据判断“失败/成功”。

---

### 2. Debug规则（先定位，后修复）
1. Debug默认只做“定位”，不得直接修改或修复。
2. 必须输出：
   - error_stage（Planner / Executor / Input / Prompt）
   - reason（基于证据）
3. 若证据不足 → 输出“UNKNOWN”，禁止猜测。

---

### 3. 修复闭环（防止盲修）
1. 修复必须基于已确认的问题来源。
2. 修复后必须回到测试 Agent 验证。
3. 未通过测试，不得进入下一阶段。

---

### 4. 失败控制（防止死循环）
1. 同一问题连续2次失败：
   - 禁止继续沿原判断修复
   - 必须返回项目总经理重新定位
2. 标记为“需要重新拆解任务或调整方案”

---

## 五、系统目标
确保系统具备：
- 可观测性（每步可追踪）
- 可验证性（有测试标准）
- 可回溯性（有证据链）
- 防幻觉（禁止自我宣称成功）
